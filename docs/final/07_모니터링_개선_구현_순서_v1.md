# 모니터링 개선 구현 순서 v1

## 목적

이 문서는 모니터링 주기, 날짜 기준, 보지 않은 기사 표시, 요약 알림 구조를 현재 `NEW\source` 기준으로 안전하게 개선하기 위한 구현 순서를 고정한다.

이번 문서에서 확정된 내용은 이후 구현과 검증의 기준으로 사용한다.

## 이번 개선의 핵심 목표

1. 전역 20분 일괄 모니터링을 카테고리별 개별 간격 구조로 전환한다.
2. `안읽음만 보기` 명칭은 `보지 않은 기사`로 통일한다.
3. 해당 날짜 최초 조회 시점 이후 자동 모니터링으로 새로 수집된 기사만 `보지 않은 기사`로 표시한다.
4. 읽음 상태와 날짜 기준선은 PC별 독립 저장으로 관리한다.
5. 윈도우 알림은 개별 폭탄 알림이 아니라 2~3분 묶음 요약 알림으로 전환한다.
6. 날짜 계산은 UTC 기준 문자열이 아니라 로컬 날짜 기준으로 통일한다.
7. 오전 9시 이전에 실행해도 조회창 기본 날짜가 항상 실제 오늘 날짜로 보이게 수정한다.

## 확정 사항

- 카테고리별 모니터링 간격 설정을 지원한다.
- 입력 방식은 `10/20/30/60분 선택 + 직접입력 허용`으로 한다.
- 읽음/보지 않음 상태는 직원별 공용이 아니라 각 PC별로 저장한다.
- 알림 클릭 시 `오늘 + 전체 카테고리 + 보지 않은 기사 보기` 상태로 이동한다.
- 추가 권장사항은 모두 포함한다.
- 이번 구현에서는 목표 설명서 밖의 기능 확장은 하지 않는다.

## 구현 원칙

### 1. 날짜 기준 원칙

- `new Date().toISOString().split('T')[0]` 방식은 전면 제거한다.
- 화면 초기 날짜, 오늘 여부 판정, 자동 모니터링 기준, 보지 않은 기사 판정은 모두 동일한 로컬 날짜 헬퍼를 사용한다.
- 날짜 기준은 PC 시스템 시간과 로컬 타임존을 따른다.

### 2. 상태 저장 원칙

- 기사 원본 데이터와 사용자 상태를 분리한다.
- 사용자 상태에는 최소한 아래 정보가 필요하다.
  - 날짜별 최초 조회 기준선 시각
  - 기사별 읽음 여부
  - 카테고리별 모니터링 간격
  - 알림 관련 기본 설정
- 사용자 상태는 PC 로컬 저장으로 유지한다.

### 3. 스케줄링 원칙

- 전역 `20분마다 전체 사이트 실행` 구조를 폐기한다.
- 내부적으로는 짧은 주기 tick으로 돌면서, 카테고리별 다음 실행 시각이 된 항목만 순차 처리한다.
- 여러 카테고리가 동시에 도래해도 병렬 폭주시키지 않고 큐 기반으로 순서대로 처리한다.
- 각 카테고리의 마지막 실행 시각, 다음 예정 시각, 최근 오류 상태를 추적 가능하게 한다.

### 4. 알림 원칙

- 알림은 2~3분 누적 후 1회 요약형으로 보낸다.
- 시스템 팝업 본문은 `카테고리명 + 건수` 중심으로 짧게 유지한다.
- 상세 제목 나열은 앱 내부 화면에서 확인하게 한다.

## 구현 순서

### 단계 1. 날짜 기준 정상화

목표:
- 앱 어디서 실행하든 조회창 기본 날짜가 로컬 오늘 날짜로 맞게 보이게 한다.
- 오전 9시 이전 실행 시 어제 날짜가 뜨는 현상을 제거한다.

작업:
- 화면 초기 날짜 계산부 교체
- 오늘 여부 비교 로직 교체
- 자동 갱신 로직의 오늘 판정 교체
- 보지 않은 기사 계산의 날짜 비교 기준 교체

완료 조건:
- 한국 시간 오전 8시에도 기본 날짜가 오늘로 표시된다.
- 과거 날짜 화면과 오늘 화면의 분기 동작이 일관된다.

### 단계 2. 사용자 상태 저장 구조 추가

목표:
- 기사 데이터와 별개로 PC별 사용자 상태를 안정적으로 저장한다.

작업:
- 날짜별 최초 조회 기준선 저장 구조 추가
- 기사별 읽음 상태 저장 구조 추가
- 카테고리별 모니터링 설정 저장 구조 추가
- 기존 UI prefs 저장 구조와 충돌 없이 분리 또는 확장

완료 조건:
- 앱 재시작 후에도 기준선, 읽음 상태, 모니터링 설정이 유지된다.

### 단계 3. 보지 않은 기사 규칙 교체

목표:
- 현재의 `lastSeenAt` 1개 기반 추정 로직을 제거하고, 합의한 규칙으로 교체한다.

작업:
- 특정 날짜를 사용자가 최초 조회한 시점을 기준선으로 저장
- 그 시점까지 존재하던 글은 일반 상태로 처리
- 이후 자동 모니터링으로 새로 수집된 글만 `보지 않은 기사` 처리
- 기사 열람 시 읽음 처리
- `보지 않은 기사` 필터 추가
- `오늘 기준선 다시 잡기` 기능 추가

완료 조건:
- 첫 조회 직후에는 기존 글에 빨간 표시가 없어야 한다.
- 그 뒤 새로 들어온 글만 표시된다.

### 단계 4. 카테고리별 모니터링 설정 UI 추가

목표:
- 설정 화면에서 카테고리별 간격을 직접 관리할 수 있게 한다.

작업:
- 카테고리별 간격 입력 UI 추가
- `10/20/30/60분` 빠른 선택 추가
- 직접입력 허용
- 카테고리별 사용/미사용 토글 검토 및 반영
- 현재 하드코딩된 `20분마다 자동 갱신` 문구 제거

완료 조건:
- 사용자가 카테고리별로 서로 다른 간격을 저장할 수 있다.
- 저장값이 앱 재시작 후에도 유지된다.

### 단계 5. 서버 스케줄러를 카테고리 순차 실행 구조로 변경

목표:
- 전체 사이트 일괄 실행을 없애고 카테고리별 due 기반 실행으로 바꾼다.

작업:
- 현재 전역 interval job 구조 분석 후 교체
- 카테고리별 다음 실행 시각 계산
- 카테고리 단위 또는 카테고리 내부 사이트 단위 순차 실행
- 중복 실행 방지
- 마지막 실행 시각, 다음 실행 시각, 오류 기록 반영

완료 조건:
- 서로 다른 카테고리가 설정된 간격대로 독립 실행된다.
- 한 시점에 알림과 크롤링이 몰리지 않는다.

### 단계 6. 요약 알림 구조로 전환

목표:
- 새 글 알림이 한 번에 쏟아지지 않도록 제어한다.

작업:
- 2~3분 누적 버퍼 추가
- 카테고리별 건수 요약 메시지 생성
- 알림 클릭 시 `오늘 + 전체 카테고리 + 보지 않은 기사 보기` 상태 진입
- 테스트 알림 기능도 새 포맷에 맞춰 정리

완료 조건:
- 짧은 시간 안의 신규 기사들이 1회 요약 알림으로 정리된다.
- 시스템 팝업은 짧고 누락 없이 동작한다.

### 단계 7. 운영성 UI 추가

목표:
- 유지보수와 현업 사용 편의를 높인다.

작업:
- 카테고리별 마지막 수집 시각 표시
- 카테고리별 다음 예정 시각 표시
- 최근 오류 여부 또는 오류 횟수 표시
- `보지 않은 기사` 필터와 `전체 보기` 전환 제공

완료 조건:
- 사용자가 현재 모니터링 상태를 설정 화면 또는 대시보드에서 확인할 수 있다.

## 검증 순서

1. 오전 9시 이전 로컬 시간 시나리오에서 기본 날짜가 오늘로 보이는지 확인
2. 오늘 날짜 최초 조회 직후 기존 글이 일반 상태인지 확인
3. 최초 조회 이후 자동 모니터링으로 들어온 글만 `보지 않은 기사`로 보이는지 확인
4. 기사 열람 후 읽음 처리되는지 확인
5. `보지 않은 기사` 필터가 정확히 동작하는지 확인
6. 카테고리별 간격을 다르게 저장하고, 실제 스케줄이 그 설정을 따르는지 확인
7. 동시에 여러 카테고리 신규 글이 생겨도 요약 알림 1회로 정리되는지 확인
8. 알림 클릭 시 전체 카테고리 `보지 않은 기사` 화면으로 이동하는지 확인
9. 앱 재시작 후 설정과 상태가 유지되는지 확인

## 구현 중 금지 사항

- 날짜 처리에서 UTC 문자열 잘라쓰기 재사용 금지
- 기존 일괄 모니터링 구조와 신규 카테고리 구조를 애매하게 혼용 금지
- 읽음 상태를 다시 단일 timestamp 추정 방식으로 되돌리는 것 금지
- 검증 없이 배포본에 바로 반영 금지

## 구현 완료 후 문서 반영 대상

- `PATCH_NOTES.md`
- 유지보수 체크리스트
- 배포/업데이트 현황 문서
- 필요한 경우 운영 기준 문서

## 비고

이번 개선은 UI 문구 변경 수준이 아니라, 날짜 기준, 사용자 상태, 스케줄링, 알림 구조를 함께 정리하는 중간 규모 개선 작업이다.

따라서 구현은 반드시 `단계별 적용 -> 단계별 검증 -> 문서 갱신` 순서로 진행한다.
